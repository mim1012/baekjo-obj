# 백조오브제 관리자 운영 매뉴얼 — 주문·결제·배송

> 대상: 관리자 콘솔(`/admin`)을 운영하는 고객사 담당자.
> 목적: 화면에서 "왜 이 버튼이 안 되지?"가 나오는 지점은 전부 **의도된 보호 장치**입니다.
> 이 문서는 각 시나리오에서 무엇을 누르고, 시스템이 무엇을 막고, 왜 막는지를 설명합니다.
> 기준일: 2026-07-18 (결제·배송 보안 하드닝 반영판). 기술 근거는 각 절 끝의 괄호 참조.

## 0. 상태값 한눈에 보기

| 결제 상태 | 의미 | 누가 만드나 |
|---|---|---|
| 결제대기 | 카드 결제 진행 중(고객이 결제창 단계) | 시스템 |
| 입금대기 | 무통장입금 주문 — 입금 확인 전 | 시스템 |
| 승인중 | 카드 승인 처리 중(순간적) | 시스템(토스) |
| **결제완료** | 돈이 실제로 확인된 유일한 상태. 매출 집계 기준 | 관리자(입금 확인) 또는 토스 |
| 결제취소 | 취소 종결 — 되돌릴 수 없음 | 취소 처리 |
| 환불완료 | 환불 종결 — 되돌릴 수 없음 | 관리자(환불 처리) |

배송 상태는 `배송전 → 배송준비 → 배송중 → 배송완료`이며, 그 뒤의 **구매확정**은
고객 버튼 또는 자동확정만이 만드는 종결 상태입니다(관리자가 직접 세팅 불가 — §4).

## 1. 무통장입금 주문 처리

1. 신규 무통장 주문은 **입금대기**로 들어오고, 그 시점에 재고가 선점됩니다.
2. 입금을 확인하면 주문 관리에서 **입금 확인**(입금대기 → 결제완료)을 누릅니다.
3. **시한 내 입금 확인이 없으면 시스템이 자동으로 주문을 취소하고 재고를 복원합니다.**
   - 기본 시한: **72시간**. 변경: 사이드바 **주문 정책**(`/admin/order-policy`)에서 1~720시간 범위로 설정.
   - 시한 변경은 **변경 이후 새로 생성되는 주문부터** 적용됩니다. 이미 들어온 주문의 만료 시각은
     생성 시점 값이 유지됩니다.
   - 자동취소된 주문은 되살릴 수 없습니다 — 고객에게 재주문을 안내하세요.

(근거: `src/lib/orderPolicy/*`, `src/lib/orders/reservationExpiry.ts`, reclaim-stock 크론)

## 2. 환불 처리

- **결제완료** 상태의 주문만 **환불완료**로 바꿀 수 있습니다.
- 환불완료는 **종결 상태**입니다. 되돌리기·재환불이 불가하며, 재고는 자동 복원되지 않습니다
  (반품 입고 후 재고는 상품 관리에서 수동 조정).

## 3. 결제 상태에서 "안 되는 것" — 전부 의도된 차단입니다

관리자가 수동으로 바꿀 수 있는 결제 상태 전이는 **딱 두 가지**입니다:

| 허용 전이 | 용도 |
|---|---|
| 입금대기 → 결제완료 | 무통장 입금 확인 |
| 결제완료 → 환불완료 | 환불 처리 |

그 외 전이는 서버가 거부합니다(화면에 오류 표시). 대표적으로:

- **아무 주문이나 결제완료로 세팅** ❌ — 결제 증빙(결제키) 없는 매출 위조를 막습니다.
- **취소했던 주문을 입금대기로 되돌리기** ❌ — 되돌린 뒤 다시 취소하면 재고가 두 번 복원돼
  실재고보다 부풀려지는 사고를 막습니다.
- **결제대기·승인중 수동 세팅** ❌ — 카드 결제 진행 상태는 결제사(토스)와 동기화되는 자동 상태입니다.

(근거: `src/lib/orders/paymentTransition.ts` 화이트리스트 + 동시수정 감지 CAS)

## 4. 배송 처리와 구매확정

- 배송 상태(배송준비/배송중/배송완료)와 송장번호는 주문 관리에서 변경합니다.
  배송 추적은 스마트택배 연동으로 자동 갱신됩니다.
- **구매확정은 관리자가 만들 수 없습니다.** 고객의 확정 버튼 또는 배송완료 후 자동확정만 가능합니다.
- **구매확정된 주문은 어떤 상태로도 되돌릴 수 없습니다(취소 기능 없음 — 2026-07-18 운영 결정).**
  확정 이후 문제 제기는 상태 되돌리기가 아니라 **환불 절차(§2)** 로 처리하세요.
- **결제되지 않은 주문은 구매확정이 되지 않습니다** — 미결제 주문이 배송완료 처리돼도
  자동확정 대상에서 걸러집니다.

(근거: `updateShipmentUnlessConfirmed` 조건부 갱신, `decideShipmentConfirm` 결제 게이트)

## 5. 자동으로 돌아가는 것들 (손대지 않아도 됩니다)

| 자동 동작 | 조건 | 결과 |
|---|---|---|
| 카드 미결제 회수 | 카드 주문이 10분 내 결제 미완료 | 주문 취소 + 재고 복원 |
| 무통장 미입금 회수 | 설정 시한(기본 72h) 내 입금 확인 없음 | 주문 취소 + 재고 복원 |
| 주문 도배 차단 | 같은 IP가 1분에 5건 초과 주문 시도 | 6번째부터 거부(잠시 후 재시도 안내) |

## 6. 자주 묻는 질문

- **Q. 입금 확인 버튼을 눌렀는데 "이미 처리됨" 오류가 떠요.**
  다른 담당자(또는 자동취소)가 먼저 처리한 것입니다. 새로고침 후 현재 상태를 확인하세요 —
  동시 수정을 시스템이 감지해 나중 조작을 거부한 것으로, 데이터는 안전합니다.
- **Q. 72시간이 너무 짧아요/길어요.** `/admin/order-policy`에서 직접 조정 가능합니다(1~720시간).
- **Q. 구매확정을 실수로 눌렀다는 고객이 있어요.** 확정 해제는 불가합니다. 환불 절차로 안내하세요.
