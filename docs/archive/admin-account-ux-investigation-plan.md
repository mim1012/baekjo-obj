# 관리자 계정·주문·배송지·진열 UX 조사 및 추진 계획

작성일: 2026-07-16  
대상: 백조오브제 관리자/마이페이지/셀렉션/주문·결제 UX

## 1. 조사 요약

### 1-1. 관리자 계정

현재 세션 기록 기준 관리자 계정은 다음과 같습니다.

- 이메일: `admin@naver.com`
- 비밀번호: `admin1234`

운영 전에는 반드시 관리자 비밀번호 교체가 필요합니다.

### 1-2. 관리자/일반회원 프로필 버튼 정책

현재 상단에는 로그인 사용자 공통 프로필 아이콘과, 관리자에게만 보이는 `관리자` 버튼이 함께 노출됩니다. 이 경우 **관리자 프로필 아이콘을 마이페이지로 보내는 것은 맞는 선택**입니다.

**이유**

- `관리자` 버튼은 업무 콘솔 진입점입니다.
- `프로필` 버튼은 “내 계정/내 주문/내 활동” 진입점입니다.
- 관리자는 관리자이면서 동시에 회원 계정이므로, 프로필 버튼까지 관리자 페이지로 보내면 개인 계정 정보·주문내역·관심상품·문의/후기 접근점이 사라집니다.
- 두 버튼이 같은 곳으로 가면 중복 CTA가 되어 UX가 나빠집니다.

**권장 정책**

- 관리자 버튼: `/admin`
- 프로필 버튼: `/mypage`
- 프로필 드롭다운 또는 마이페이지 내부 기본 강조: `주문내역`
- 모바일 하단 “마이”: `/mypage`
- 주문내역 바로가기가 필요하면 프로필 드롭다운 안에 “주문내역” 항목을 별도로 제공

즉, **관리자에게도 프로필 버튼은 마이페이지로 유지**하고, 관리자 콘솔은 별도 관리자 버튼으로 분리하는 것이 맞습니다.

### 1-3. 셀렉션 페이지 하트 버튼

**사용자 피드백: 하트 버튼은 제거.**

현재 하트 버튼은 브라우저 `localStorage`의 `baekjo_wishlist`에만 저장되는 관심 상품/찜 기능입니다. 계정, DB, 관리자 화면과 동기화되지 않아 고객이 “저장됐다”고 기대하기 어렵습니다. 따라서 현 단계에서는 DB 동기화를 새로 추진하기보다 **셀렉션/상품 상세의 하트 버튼을 제거**하는 방향이 맞습니다.

**정리 방향**

- 셀렉션 상품 카드의 하트 버튼 제거
- 상품 상세의 찜 버튼 제거
- 마이페이지 `관심 상품` 탭은 함께 숨기거나 제거 검토
- 기존 `localStorage` wishlist 데이터는 당장 마이그레이션하지 않음
- 향후 찜/관심상품을 다시 열 때는 로그인 회원 DB 동기화까지 포함해 별도 기능으로 재설계


### 1-4. 브랜드관에서 상품 상세 이동 시 404

**사용자 피드백: 비노출 상품은 브랜드관에도 보이면 안 됩니다.**

현재 문제의 본질은 “비노출 상품 클릭 시 어디로 보낼 것인가”가 아니라, **공개 브랜드관이 관리자 권한일 때 비노출 상품까지 섞어 보여주는 설계 자체가 잘못된 것**입니다. 공개 브랜드관은 로그인 역할과 무관하게 고객이 보는 공개 화면이어야 합니다.

현재 구조에서는 브랜드 상세 화면이 최초 서버 렌더링 때는 공개 상품만 받지만, 클라이언트에서 관리자/파트너 권한을 확인하면 비노출 포함 전체 상품으로 목록을 교체합니다. 그 결과 브랜드관에서는 비노출 상품이 보이고, 상품 상세 `/shop/[id]`에서는 비노출 정책 때문에 404가 나는 모순이 생깁니다.

**수정 방향**

- 공개 브랜드관(`/brands/[id]`)은 관리자/파트너로 로그인해도 `isVisible=true` 상품만 표시
- 비노출 상품은 브랜드관에 표시하지 않음
- 비노출 상품 관리는 `/admin/products`, `/admin/products/[id]`, `/admin/products/display`에서만 수행
- 공개 상품 상세(`/shop/[id]`)의 visible-only 정책은 유지
- 관리자 미리보기가 필요하면 공개 브랜드관에 섞지 말고, 별도 “관리자 미리보기/편집” 동선으로 분리

즉, **브랜드관과 상품 상세 모두 같은 공개 노출 기준을 써야 하며, 비노출 상품을 관리자 상세로 우회 이동시키는 방식은 채택하지 않습니다.**

### 1-5. 비노출 상품 전역 노출 원칙

**사용자 피드백: 비노출 상품은 브랜드관뿐 아니라 고객이 보는 어떤 상품 노출면에도 나오면 안 됩니다.**

이 판단이 맞습니다. 상품의 `isVisible=false`는 “상세 페이지만 404”가 아니라 **공개 카탈로그 전체에서 제외**라는 뜻이어야 합니다. 특정 페이지에서는 보이고 다른 페이지에서는 안 보이면 비노출 기능의 의미가 사라집니다.

**공개 노출면에서 반드시 제외**

- 홈 추천/베스트/브랜드 섹션
- 셀렉션(`/shop`) 목록/검색/필터/추천 영역
- 브랜드관(`/brands`, `/brands/[id]`) 상품 영역
- 진단 결과 추천 상품
- 고민/콘텐츠 기반 추천 상품
- 장바구니 재조회 및 checkout 진입
- 공개 상품 상세(`/shop/[id]`)

**예외적으로 보여도 되는 곳**

- 관리자 상품 관리/진열 관리
- 파트너 또는 관리자 전용 상품 관리 화면
- 이미 구매한 회원의 주문내역/영수 기록에 필요한 최소 상품 정보

마지막 예외는 “판매 노출”이 아니라 **거래 기록 보존**입니다. 사용자가 과거에 산 상품이 이후 비노출 처리됐다고 해서 주문내역에서 상품명까지 사라지면 CS/환불/후기 관리가 더 깨집니다. 단, 여기서도 재구매 CTA나 공개 상세 이동은 막아야 합니다.

**점검 방향**

- 공개 화면은 항상 `getPublicProducts` 또는 `listProducts({ visibleOnly: true })`만 사용
- `includeHidden: true`, `listAllProductsForAdmin`, `getPartnerProducts`는 관리자/파트너 관리 화면과 구매 이력 복원 전용으로만 사용
- 비노출 상품이 공개 카드 컴포넌트에 들어오지 않도록 서버/콘센트 단계에서 차단

## 2. 상품 진열 및 카테고리 관리 UIUX 피드백

### 2-1. 현재 구조

관리자 상품 진열 관리는 다음 탭으로 구성되어 있습니다.

- 베스트 상품
- 추천 상품(MD)
- 스토어 노출 상태

카테고리 관리는 다음 항목을 편집합니다.

- 상품 카테고리
- 라이프스타일 카테고리

### 2-2. 주요 UX 문제

1. **드래그 정렬처럼 보이나 실제 드래그 정렬이 아님**
   - 진열 순서 관리 기능으로 오해할 수 있습니다.

2. **상품 상태가 한눈에 보이지 않음**
   - 가격 미등록
   - 품절
   - 비노출
   - 브랜드 없음
   - BEST/추천 여부
   - 위 상태들이 상품 리스트에서 명확하지 않습니다.

3. **현재 등록/추가 가능 목록의 의미가 약함**
   - 초보 관리자는 어느 쪽이 실제 고객 화면에 노출되는 목록인지 즉시 이해하기 어렵습니다.

4. **스토어 노출 해제는 위험 작업인데 확인 흐름이 약함**
   - 상품 숨김은 고객 화면과 진단 추천 결과에 영향을 줄 수 있습니다.

5. **카테고리 관리 범위가 불완전함**
   - 타입에는 `brandFilters`가 있으나 현재 UI에서는 관리하지 않습니다.

6. **노출/비노출 필터는 상품 관리에는 있으나, 진열 관리 UX와 전역 노출 정책까지 연결되지 않음**
   - 상품 관리 목록에는 `노출 상태` 필터가 이미 있습니다.
   - 다만 “숨김 상품만 보기”, “노출 상품만 보기”, “공개 화면에서 실제로 제외되는지 확인”이 운영 흐름으로 충분히 강조되지 않습니다.
   - 상품 진열 관리 화면에서도 추가 가능한 상품 목록에 비노출 상품이 섞이면 “진열 가능”으로 오해할 수 있습니다.

### 2-3. 개선 방향

- 상품 관리 상단에 노출 상태 요약 카드 추가
  - 전체 상품
  - 노출 상품
  - 비노출 상품
  - 가격 미등록
  - 품절
  - 상세 미작성
- 노출/비노출 필터를 더 전면화
  - `전체`, `노출`, `비노출` 세그먼트 탭 또는 칩으로 제공
  - 현재 select 필터는 유지하되, 운영자가 자주 쓰는 상태는 원클릭 필터로 노출
- 비노출 상품 행은 시각적으로 구분
  - 낮은 opacity만 쓰지 말고 `비노출` 배지와 “공개 화면 전체에서 제외됨” 설명 제공
- 상품 진열 관리에서는 비노출 상품을 기본적으로 “추가 가능” 후보에서 제외
  - 필요 시 `비노출 포함` 토글을 켰을 때만 보이게 함
  - 비노출 상품을 베스트/추천으로 추가하려 하면 “먼저 노출 처리 필요” 안내
- 좌측: 현재 진열/노출 목록
- 우측: 추가 가능한 상품 검색/필터
- 상품 카드에 상태 배지 표시
  - `노출`, `비노출`, `가격 미등록`, `품절`, `BEST`, `추천`
- 노출 해제 시 확인 모달 제공
- 변경 저장 전 요약 표시
  - 예: “3개 추가, 1개 해제, 2개 순서 변경”
- 진열 순서가 필요하면 실제 drag-sort 구현
- “상품 진열 관리”와 “카테고리 관리” 사이 관계 설명 추가

## 3. 배송지 개선 조사

### 3-1. 현재 checkout 배송지 필드

현재 주문/결제 화면에는 다음 필드만 있습니다.

- 받는 사람
- 연락처
- 주소
- 배송 메모

도로명주소 검색, 우편번호, 상세주소 분리, 배송지 저장/추가/삭제 기능은 없습니다.

### 3-2. 배송지 필수값 권장

네이버쇼핑/쿠팡 수준의 기본 배송지 UX를 기준으로 다음 필드를 권장합니다.

**필수**

- 수령인
- 휴대폰 번호
- 우편번호
- 도로명주소
- 상세주소

**선택 또는 조건부**

- 배송메모
- 배송지명: 집, 회사, 직접입력
- 기본 배송지 여부
- 공동현관 출입방법/비밀번호
- 부재 시 요청사항

### 3-3. 도로명주소 검색 방식

권장 후보는 다음과 같습니다.

1. **Kakao/Daum 우편번호 서비스**
   - 문서: https://postcode.map.kakao.com/guide
   - 반환값: `zonecode`, `roadAddress`, `jibunAddress`, `buildingName` 등
   - 모바일에서는 팝업보다 레이어/iframe 방식 권장

2. **도로명주소 안내시스템/주소정보누리집 API**
   - 문서: https://juso.go.kr/
   - 공공 도로명주소 검색 API 계열

**권장안**

- 1차 구현은 Kakao/Daum 우편번호 서비스 사용
- 선택 결과는 `zonecode`, `roadAddress`, `jibunAddress`로 저장
- 상세주소는 별도 필수 입력
- 향후 필요 시 공공 주소 API로 교체 가능하게 주소 검색 컴포넌트 분리

### 3-4. 배송지 저장/관리 UX

회원 기준:

- 배송지 목록 보기
- 배송지 추가
- 배송지 수정
- 배송지 삭제
- 기본 배송지 설정
- checkout 진입 시 기본 배송지 자동 채움

비회원 기준:

- 주문 중 1회성 배송지 입력
- 회원가입/로그인 유도는 결제 흐름을 막지 않는 보조 CTA로만 제공

## 4. 무통장입금 UX 및 관리자 관리

### 4-1. 현재 상태

현재 무통장입금 선택 후 주문 생성이 완료되면 `/order-complete`로 이동합니다. 별도의 계좌 안내 카드나 관리자 계좌 설정 기능은 없습니다.

### 4-2. 개선 방향

무통장입금은 `/order-complete`로 단순 이동시키기보다, checkout 흐름 안에서 반응형 입금 안내를 즉시 보여주는 방식이 좋습니다.

**고객 화면**

- 무통장입금 선택 시 계좌 안내 카드 표시
- 주문 접수 후 “입금 대기 주문이 접수되었습니다” 패널 표시
- 계좌번호 복사 버튼 제공
- 입금자명 안내
- 주문 금액/입금 기한/입금 확인 안내 표시

**관리자 화면**

신규 메뉴 또는 환경 설정 내 “결제 설정” 추가:

- 무통장입금 사용 여부
- 은행명
- 계좌번호
- 예금주
- 안내 문구
- 입금 기한 안내
- 주문자명과 입금자명이 다를 때 안내

## 5. 주문번호 개선

현재 주문 ID는 DB UUID라 고객과 관리자 화면에 너무 길게 보입니다.

**개선 방향**

- 내부 PK: UUID 유지
- 표시용 주문번호 별도 추가
- 예시:
  - `BO-260716-1042`
  - `260716-AB12`
  - `BO20260716-0001`

표시용 주문번호는 고객 화면, 마이페이지, 관리자 주문관리, 입금 안내에 사용하고, UUID는 내부 조회/보안/개발용으로 유지합니다.

## 6. 주문관리 금액 불일치 원인

사용자 사례: 8,900원 결제했는데 주문관리 페이지에는 5,900원이 표시됨.

조사 결과, 원인은 거의 확정적입니다.

- 주문 저장 구조:
  - `totalPrice`: 상품금액
  - `deliveryFee`: 배송비
- checkout/order-complete 표시:
  - `totalPrice + deliveryFee`
- 관리자 주문관리 목록/상세 일부 표시:
  - `totalPrice`만 표시

즉, 상품 5,900원 + 배송비 3,000원 = 최종 8,900원인데, 관리자에는 상품금액 5,900원만 “총 결제금액”처럼 보이는 상태입니다.

**즉시 수정 필요**

- 관리자 주문 목록: 최종 결제금액은 `totalPrice + deliveryFee`
- 관리자 주문 상세: 상품금액/배송비/최종결제금액 분리 표시
- 모바일 주문 카드도 동일하게 수정

## 10. 결론

가장 먼저 고쳐야 할 것은 **주문관리 금액 불일치**입니다. 실제 결제/입금 확인 운영에 직접 영향을 주며, 관리자 신뢰도를 떨어뜨립니다.

그다음은 **비노출 상품 전역 노출 점검 및 제거**, **관리자/프로필 버튼 역할 분리 확인**, **하트 버튼 제거**입니다. 셋 다 계약 변경 없이 빠르게 고칠 수 있습니다.

배송지, 무통장입금 계좌 관리, 짧은 주문번호는 계약/DB 변경이 필요한 구조 개선이므로 별도 PR로 분리하는 것이 안전합니다. 관심상품은 현 단계에서 확장하지 않고 제거/숨김 처리 후, 추후 필요해질 때 DB 기반으로 다시 설계합니다.
