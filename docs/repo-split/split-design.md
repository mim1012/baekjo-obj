## 0. 구조

| 레포 | 소유 | 성격 | 의무 |
|------|------|------|------|
| `baekjo-design` (신규) | **dad** | UI 전시장. mock으로 도는 Next.js 앱. dad가 원하는 대로 쌓는다 | **거의 없음** (아래 "부탁 3개"뿐) |
| `baekjo-obj` (현행) | **mim** | 프로덕션 정본. 수확·배선·검증·배포 전부 여기 | 수확 프로토콜(§2) 수행 |

흐름은 **단방향 1개**:

```
dad 레포 (UI 정본, 자유 게시)  ──▶  obj (우리가 수확 → 배선 → 검증 → 배포)
```

- v1의 "contract sync(obj→design) 의무"는 **폐지**. dad는 계약 파일을 신경 쓰지 않는다.
- 대신 우리가 **선택적으로** 최신 types/mock 스냅샷을 dad 레포에 밀어줄 수 있다(§3) —
  dad 화면이 실데이터 모양에 가까울수록 수확이 싸지기 때문. 의무가 아니라 우리 편의.

## 1. dad 쪽 — 자유, 단 "부탁 3개"

dad 레포엔 규칙이 아니라 **가져가기 쉽게 해주는 부탁 3개**만 둔다
(상세: [design-repo-AGENTS-draft.md](./design-repo-AGENTS-draft.md)):

1. **`npm run build`가 깨지지 않은 상태로 올려달라** — 깨진 트리는 어디까지가 의도인지 알 수 없어 수확 불가.
2. **데이터는 컴포넌트 props나 상단 mock 상수로 받게 만들어달라** — 컴포넌트 깊숙이 하드코딩된
   텍스트/배열은 배선 시 한 줄씩 파내야 한다. props/상수면 교체가 기계적이다.
3. **파일 위치는 지금 구조(`src/components/**`, `src/app/**`)를 유지해달라** — 경로가 같아야
   diff로 "무엇이 바뀌었나"를 자동 추출할 수 있다.

이 3개가 지켜지는 만큼 수확이 싸진다. 안 지켜져도 수확은 가능하다 — 우리 비용이 늘 뿐.

## 2. 우리 쪽 — 수확 프로토콜 (obj AGENTS.md §8-1을 대체할 내용)

### 2-1. 수확 단위와 주기

- 단위: **화면(페이지) 1개 또는 컴포넌트 군 1개** = 수확 브랜치 1개(`fe/design-sync-<대상>-<날짜>`).
  dad 레포 전체를 한 방에 가져오지 않는다(b85e723 대형 통합 사고의 교훈 — 크면 유실을 못 본다).
- 주기: dad가 "이거 봐줘/올렸어" 신호를 주면, 또는 주 1회 dad 레포 diff를 훑고 수확 대상 선정.

### 2-2. 수확 절차 (스크립트 `scripts/design-harvest.ps1`로 공식화)

```
① diff 추출: dad 레포의 전회 수확 지점(태그 harvested/<날짜>) 이후 변경 파일 목록
② 분류:
   - 표현 파일(components·*Client.tsx·globals.css·public·정적콘텐츠 data) → 그대로 복사
   - page.tsx·mock 데이터(products/brands류) 변경 → 복사하지 않고 "배선 필요" 목록으로 출력
③ obj 수확 브랜치에 커밋 — dad 저작자 보존(author 지정 또는 Co-authored-by)
④ 배선: dad가 mock으로 그린 데이터 흐름을 콘센트(storage)/repo로 재배선.
   ⭐ 정본 규칙 유지: 마크업·스타일·섹션 배치·순서 = 100% dad. 우리는 import·props만 만진다.
⑤ 검증: build + lint(§4-6이 mock import 잔재 차단) + visual 14장(빨간불=정상 → diff 확인 →
   update-baselines) + 해당 골든플로우 스모크
⑥ 머지 후 dad 레포에 태그 harvested/<날짜> — 다음 수확의 diff 기준점
```

### 2-3. dad가 새 데이터 필드를 쓴 경우 (수확에서 가장 흔한 갈래)

dad는 계약을 모른 채 mock 필드를 마음대로 만든다 — **그래도 된다.** 처리:

1. 수확 ④에서 dad mock의 모양과 우리 `types/index.ts`의 갭을 목록화.
2. 갭이 가산적(필드 추가)이면: 같은 수확 PR에서 optional 필드로 계약 확장 + seed 반영(§4 규칙대로).
3. 갭이 파괴적(기존 필드 의미 변경)이면: 수확 PR을 표현만으로 줄이고, 계약 변경은
   별도 contract PR로 선행(§0-2 ②). 그동안 해당 화면은 기존 필드로 폴백 배선.

### 2-4. 유실 방지 (v1의 해시 하드스톱 대체)

- v1의 "계약 해시 불일치 시 이식 중단"은 dad에게 동기화 의무를 지우므로 폐지.
- 대신 유실 감지는 **obj 쪽 게이트가 전담**: 수확 PR의 visual diff를 dad가 프리뷰로 직접 확인
  ("내가 올린 그대로인가")하는 것을 머지 조건으로. dad의 리뷰는 코드가 아니라 화면(§8-1 기존 원칙 그대로).
