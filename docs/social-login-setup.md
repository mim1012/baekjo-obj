## 2. 환경변수 (콘센트에 꽂을 키)

`.env.example` 참고. 로컬은 `.env.local`에, 운영은 Vercel 환경변수에 넣는다.

| 변수 | 값 출처 |
|------|---------|
| `AUTH_SECRET` | `npx auth secret` 한 번 실행해서 생성 (아무 랜덤 32바이트) |
| `AUTH_KAKAO_ID` | 카카오 디벨로퍼스 → 앱 → **REST API 키** |
| `AUTH_KAKAO_SECRET` | 카카오 디벨로퍼스 → 제품 설정 → 카카오 로그인 → 보안 → **Client Secret** (활성화 필요) |
| `AUTH_NAVER_ID` | 네이버 개발자센터 → 애플리케이션 → **Client ID** |
| `AUTH_NAVER_SECRET` | 같은 곳 → **Client Secret** |
| `NEXT_PUBLIC_SOCIAL_LOGIN` | `1`이면 로그인 화면 소셜 버튼 활성화. **키를 다 채운 뒤에만 1로.** |

## 3. 카카오 디벨로퍼스 등록 절차

1. https://developers.kakao.com → 내 애플리케이션 → **애플리케이션 추가** (앱 이름: 백조오브제)
2. 제품 설정 → **카카오 로그인 활성화** ON
3. 카카오 로그인 → **Redirect URI 등록** (둘 다):
   - `http://localhost:3000/api/auth/callback/kakao`
   - `https://baekjo-obj.vercel.app/api/auth/callback/kakao`
4. 카카오 로그인 → 보안 → **Client Secret 생성 + 활성화**
5. **동의항목**: 닉네임·프로필 이미지 = 기본 동의 설정.
   ⚠️ **이메일은 "비즈 앱" 전환(사업자등록번호 인증) 후에만 필수 수집 가능** — 그 전엔 이메일 없이 로그인될 수 있고, 코드가 이 경우를 처리함(플레이스홀더 이메일).
6. (운영 전환 시) 앱 → 비즈니스 → **비즈 앱 전환**: 사업주의 사업자등록번호 필요.

## 4. 네이버 개발자센터 등록 절차

1. https://developers.naver.com → Application → **애플리케이션 등록**
2. 사용 API: **네이버 로그인** / 제공 정보: 이메일, 이름(닉네임), 프로필 사진
3. 서비스 URL: `https://baekjo-obj.vercel.app`
4. **Callback URL 등록** (둘 다):
   - `http://localhost:3000/api/auth/callback/naver`
   - `https://baekjo-obj.vercel.app/api/auth/callback/naver`
5. ⚠️ 검수 전("개발 중" 상태)에는 **멤버 관리에 등록된 아이디만 로그인 가능** — 테스트할 계정을 멤버로 추가.
6. (운영 전환 시) **네이버 로그인 검수 신청**: 개인정보처리방침 URL 필수 → 사이트에 해당 페이지 준비 필요.

## 5. 계정 소유권 정책 (누가 등록하나)

- **최종(운영) 앱 = 사업주 계정 소유.** 카카오 비즈 앱(사업자번호)·네이버 검수가 사업주 명의 기준이고, 납품물 소유권 분쟁 방지.
- 사업주에게 요청할 것 (각 1회):
  1. 카카오 디벨로퍼스 가입 → 앱 생성 → **팀 관리에서 mim1012 카카오 계정을 관리자로 초대**
  2. 네이버 개발자센터 가입 → 앱 등록 → **멤버 관리에 mim1012 네이버 아이디를 관리자로 등록**
  - 이후 모든 설정 변경은 개발자가 직접 — 사업주에게 매번 로그인 요청할 필요 없음.
- **개발 중에는 개발자 개인 계정의 "개발용 앱"으로 진행** (키는 환경변수라 코드 변경 0줄).
  > ⚠️ **실DB(베타 회원 시스템, 2026-07-10~) 도입 이후 주의:** 이제 소셜 회원을 `(provider, provider_id)`로 식별한다.
  > `provider_id`는 **앱마다 다르게 발급**되므로, 소유자 명의로 **앱을 새로 만들어 키만 갈아끼우면** 같은 사용자가 다시
  > 로그인할 때 **다른 provider_id → 중복 회원**이 된다(이메일 없는 카카오 유저는 100% 중복). 수집한 소셜 회원을
  > 보존하려면 **같은 앱을 유지한 채 소유권만 이전**해야 한다. 상세 절차는 **§8**.

## 6. Vercel 프리뷰 배포 주의

프리뷰 URL(`baekjo-obj-git-*.vercel.app`)은 매번 바뀌어서 Redirect URI로 등록 불가.
→ **소셜 로그인 실테스트는 localhost 또는 프로덕션 도메인에서만.** 프리뷰에서는 버튼이 OAuth 단계에서 redirect_uri 오류가 나는 게 정상.

## 8. 베타 → 소유자 핸드오버 (수집한 회원 데이터 보존)

> 질문: "지금 우리(개발자) 계정으로 카카오·네이버·이메일 인증까지 켜서 베타 회원을 DB에 수집하고,
> 나중에 소유자(사업주) 것으로 바꿔도 데이터가 보존되나?" → **된다.** 단 카카오·네이버만 방식에 주의.
> 핵심: **수집한 회원은 Supabase DB에 있고, 인증 수단(카카오/네이버/메일)과 독립적으로 그대로 남는다.**

### 대상별 요약

| 대상 | 나중에 소유자 걸로 교체 | 수집 데이터 영향 | 하는 일 |
|------|------------------------|------------------|---------|
| **이메일 발송(SMTP)** | ✅ 안전 | 없음 | `SMTP_GMAIL_*` env만 소유자 계정/도메인으로 교체(또는 Resend+도메인). 발신자만 바뀜 |
| **DB(Supabase)** | ✅ 안전 | 없음 | Supabase 프로젝트를 소유자 조직으로 이관하거나 접근권만 이전. 회원 행은 그대로 |
| **카카오·네이버** | ⚠️ 방식 주의 | 앱을 새로 만들면 중복 | **같은 앱 유지 + 소유권 이전**(아래) |

### 카카오·네이버 — 반드시 "새 앱 X, 소유권 이전 O"

회원 식별 키가 `(provider, provider_id)`이고 `provider_id`는 **그 OAuth 앱이 발급한 고유 번호**다.
- ❌ **소유자 명의로 앱 신규 생성 → 키 교체**: 같은 사용자도 새 앱에선 다른 `provider_id`를 받아 **중복 회원 생성**.
  이메일 없는 카카오 유저(비즈앱 전 placeholder 이메일)는 100% 중복. 지금 모은 소셜 회원이 갈라진다.
- ✅ **지금 앱을 그대로 두고 소유권만 이전** → `provider_id` 불변 → 데이터 그대로 이어짐. `AUTH_KAKAO_ID`/
  `AUTH_NAVER_ID` 등 키도 안 바뀌므로 **env·코드 수정조차 0**.

핸드오버 실행(각 1회):
1. **카카오**: 개발용 앱 → 팀 관리에서 사업주 계정을 관리자로 추가 → (원하면) 소유권 이전. 앱 유지, 키 유지.
   비즈 앱 전환(사업자등록번호)은 이 **같은 앱**에서 진행 → 그때부터 이메일 필수 수집.
2. **네이버**: 개발용 앱 → 멤버 관리에서 사업주 아이디를 관리자로 추가 → 소유권 이전. 앱 유지, 키 유지.
   검수도 이 앱으로 신청.

> 만약 불가피하게 앱을 새로 만들어야 하는 상황이 오면: 이메일 있는 소셜 회원은 Phase 2의 "계정 연동(이메일 재매칭)"
> 플로우로 살릴 수 있으나, **이메일 없는 카카오 유저는 못 살린다.** → 그래서 같은 앱 이전이 정답.

### 이메일 회원 / DB — 그냥 넘기면 됨

- **이메일(비밀번호) 회원**: 발신 SMTP를 소유자 것으로 바꿔도 회원 레코드·`email_verified` 무영향. `mailer.ts` 한 곳만 교체.
- **Supabase**: 프로젝트 소유권 이전(Organization transfer) 또는 접근 권한만 위임. `members`·`member_tokens` 데이터는 이사 없이 그대로. `SUPABASE_*` env만 새 소유자 값으로 교체하면 됨.

### 핸드오버 체크리스트

- [ ] 카카오/네이버: **같은 앱** 소유권을 사업주로 이전(신규 생성 금지) — provider_id 보존 확인
- [ ] 카카오 비즈 앱 전환(사업자번호) → 이메일 필수 수집으로
- [ ] Supabase 프로젝트를 사업주 조직으로 이관(또는 접근권 위임) → `SUPABASE_*` env 교체
- [ ] SMTP를 사업주 계정/도메인으로 교체(또는 Resend+도메인) → `SMTP_GMAIL_*` env 교체
- [ ] `APP_BASE_URL`을 최종 운영 도메인으로 확정
- [ ] 이전 후 실계정 1회 재로그인 → 기존 회원으로 **합류**되는지(중복 생성 아님) 확인
